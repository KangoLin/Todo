# 土豆Todo — 项目功能与架构规划

> 基于调研总结报告的技术落地规划  
> 框架：Tauri v2 + React 19 + TypeScript + Rust + SQLite

---

## ⚠️ 实际项目方向说明

本规划文档最初基于完整的 Kanban 看板架构设计。实际开发中，产品方向选定为 **每日时间块规划器（Daily Time-Block Planner）**，属于看板产品的垂直切分场景。因此文档中存在以下差异：

| 规划中的概念 | 实际代码中的概念 |
|-------------|-----------------|
| Project / Board / Column | Card（便签 = 一天） |
| Card | Item（事项 = 时间块任务） |
| 多列看板 | 单列便签列表（水平排列） |
| 跨列拖拽 | 跨便签拖拽移动事项 |
| FTS5 全文搜索 | 前端简单标题过滤 |

当前代码实际构建的是 **Phase 0.5**，介于调研与完整 Phase 1 之间的最小可行产品。以下规划保留原始架构设计作为远期参考，**当前优先级以前端定制的 Phase 0.5 ~ Phase 1 为准**。

## 三、Tauri 命令（Rust → 前端 API）设计

所有前后端通信通过 Tauri `invoke` 命令完成。

### 3.1 项目管理

| 命令 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `create_project` | name, description?, color? | Project | 创建项目 |
| `get_all_projects` | — | Vec\<Project\> | 获取所有项目 |
| `get_project` | id: String | Project | 获取单个项目 |
| `update_project` | id, name?, description?, color? | Project | 更新项目 |
| `delete_project` | id: String | () | 删除项目（级联删除其下所有数据） |

### 3.2 看板管理

| 命令 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `create_board` | project_id, name | Board | 创建看板 |
| `get_boards_by_project` | project_id | Vec\<Board\> | 获取项目下所有看板 |
| `update_board` | id, name? | Board | 更新看板 |
| `delete_board` | id | () | 删除看板（级联删除列和卡片） |
| `reorder_boards` | ids: Vec\<String\> (ordered) | () | 拖拽看板重新排序 |

### 3.3 列管理

| 命令 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `create_column` | board_id, name, sort_order? | Column | 创建列 |
| `get_columns_by_board` | board_id | Vec\<Column\> | 获取看板下所有列 |
| `update_column` | id, name?, wip_limit? | Column | 更新列 |
| `delete_column` | id | () | 删除列 |
| `reorder_columns` | ids: Vec\<String\> (ordered) | () | 拖拽列重新排序 |

### 3.4 卡片管理

| 命令 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `create_card` | column_id, title | Card | 创建卡片 |
| `get_cards_by_column` | column_id | Vec\<Card\> | 获取列下所有卡片 |
| `get_card` | id | Card (with subtasks & tags) | 获取卡片完整信息 |
| `update_card` | id, title?, description?, due_date?, priority? | Card | 更新卡片 |
| `move_card` | card_id, target_column_id, target_sort_order | () | 拖拽移动卡片到目标列 |
| `move_card_within_column` | card_id, target_sort_order | () | 同列内拖拽排序 |
| `archive_card` | id | () | 归档卡片 |
| `delete_card` | id | () | 永久删除卡片 |

### 3.5 子任务

| 命令 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `create_subtask` | card_id, title | Subtask | 创建子任务 |
| `toggle_subtask` | id | Subtask | 切换子任务完成状态 |
| `delete_subtask` | id | () | 删除子任务 |

### 3.6 标签

| 命令 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `create_tag` | board_id, name, color | Tag | 创建标签 |
| `get_tags_by_board` | board_id | Vec\<Tag\> | 获取看板下所有标签 |
| `delete_tag` | id | () | 删除标签 |
| `add_tag_to_card` | card_id, tag_id | () | 给卡片添加标签 |
| `remove_tag_from_card` | card_id, tag_id | () | 从卡片移除标签 |
| `get_tags_by_card` | card_id | Vec\<Tag\> | 获取卡片上的所有标签 |

### 3.7 搜索

| 命令 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `search_cards` | query: String, project_id? | Vec\<SearchResult\> | FTS5 全文搜索 |

### 3.8 设置与工具

| 命令 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `get_setting` | key: String | String? | 获取设置 |
| `set_setting` | key, value | () | 设置配置项 |
| `export_data` | path: String | () | 导出所有数据为 JSON |
| `import_data` | path: String | () | 从 JSON 导入数据 |
| `get_db_stats` | — | DbStats | 获取数据库统计信息 |

## 四、前端路由设计

使用 React Router v7：

| 路由 | 页面组件 | 说明 |
|------|----------|------|
| `/` | 重定向到首次跳转 | 空状态引导页 |
| `/projects` | ProjectListPage | 项目列表主页 |

| `/search` | SearchPage | 全局搜索结果页 |

## 六、数据流架构

### 6.1 单向数据流

```
用户操作 (拖拽/点击/输入)
        │
        ▼
React 组件 → Zustand (UI 状态) / React Query (服务端状态)
        │
        ▼
TanStack React Query (缓存层)
        │
        ▼
tauri.invoke("command_name", { args })
        │
        ▼
Tauri Commands (Rust)
        │
        ▼
rusqlite (SQLite 数据库)
        │
        ▼
返回 Result<T, String> 给前端
        │
        ▼
React Query 自动更新缓存 → UI 重新渲染
```

### 6.2 状态管理分工

| 状态类型 | 管理工具 | 示例 |
|----------|----------|------|
| **服务端状态**（数据库数据） | TanStack React Query | 项目列表、看板数据、卡片数据 |
| **UI 状态**（无需持久化） | Zustand | 当前选中项目/看板、侧边栏展开/收起、主题模式 |
| **拖拽状态**（临时） | Zustand（或 Pragmatic DnD 内部状态） | 拖拽中的源/目标位置、动画状态 |
| **表单状态** | React 本地 state / React Hook Form | 卡片编辑弹窗的输入值 |

### 6.3 缓存策略

```
TanStack React Query:
├── queryKey: ["projects"]                    → staleTime: 5min (项目不常变)
├── queryKey: ["boards", projectId]           → staleTime: 5min
├── queryKey: ["columns", boardId]            → staleTime: Infinity (几乎不变)
├── queryKey: ["cards", columnId]             → staleTime: 0 (实时更新，拖拽频繁)
├── queryKey: ["card", cardId]                → staleTime: 0
├── queryKey: ["tags", boardId]               → staleTime: Infinity
└── queryKey: ["search", query]               → staleTime: 0 (搜索结果即时)
```

- **高频操作**（拖拽移动卡片）→ `staleTime: 0`，拖拽完成后乐观更新 + 后端确认
- **低频操作**（项目/看板元数据）→ `staleTime: 5min`，减少不必要查询
- **常量数据**（列结构、标签）→ `staleTime: Infinity`，只在变更时主动失效

```sql
);

    card_id     TEXT NOT NULL REFERENCES cards(id) ON DELETE CASCADE,
    title       TEXT NOT NULL,

    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

    UNIQUE(board_id, name)
);

    card_id UNINDEXED,
    title,
    description,
    content='cards',
    content_rowid='rowid',
    tokenize='unicode61'
);

-- 触发器：保持 FTS 索引与 cards 表同步
CREATE TRIGGER cards_ai AFTER INSERT ON cards BEGIN
    INSERT INTO cards_fts(rowid, card_id, title, description)
    VALUES (new.rowid, new.id, new.title, new.description);
END;

CREATE TRIGGER cards_ad AFTER DELETE ON cards BEGIN
    INSERT INTO cards_fts(cards_fts, rowid, card_id, title, description)
    VALUES ('delete', old.rowid, old.id, old.title, old.description);
END;

CREATE TRIGGER cards_au AFTER UPDATE ON cards BEGIN
    INSERT INTO cards_fts(cards_fts, rowid, card_id, title, description)
    VALUES ('delete', old.rowid, old.id, old.title, old.description);
    INSERT INTO cards_fts(rowid, card_id, title, description)
    VALUES (new.rowid, new.id, new.title, new.description);
END;
```

## 十、关键设计决策说明

### 10.1 为什么用 React Query 而非直接调用 Tauri 命令？

虽然 Tauri invoke 本质是异步函数，但 React Query 提供了：
- 自动缓存 + 去重（避免同一数据多次查询）
- 乐观更新（拖拽卡片后先更新 UI，再等待后端确认）
- 错误重试 + 回滚（网络/数据库错误时 UI 不崩溃）
- 数据失效 + 自动刷新（某个命令成功后，相关 query 自动刷新）

### 10.2 为什么用 Zustand 而非 Context/Redux？

- Zustand 包体积 ~1KB（Redux Toolkit ~11KB）
- 无需 Provider 包裹
- 可在组件外部使用（方便拖拽回调中访问状态）
- TypeScript 友好
- Zustand v5 支持 Signals，性能更优

### 10.3 拖拽数据持久化策略

```
用户拖拽卡片到新列
        │
        ▼
Pragmatic DnD 触发 onDrop 回调
        │
        ▼
1. 乐观更新：React Query cache 中立即更新卡片位置
2. UI 即时反馈（无等待）
        │
        ▼
3. 并行调用：tauri.invoke("move_card", { ... })
        │
        ▼
4. 成功 → React Query 刷新确认数据
5. 失败 → 回滚到之前的位置 + 错误提示
```

### 10.4 Tiptap 富文本存储格式

```json
{
  "type": "doc",
  "content": [
    {
      "type": "paragraph",
      "content": [
        { "type": "text", "text": "这是普通文本 " },

        { "type": "text", "text": " " },
```
